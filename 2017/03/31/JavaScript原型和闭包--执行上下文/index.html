JavaScript原型和闭包–执行上下文
  1. 什么是“执行上下文”（也叫做“执行上下文环境”）？暂且不下定义，先看一段代码：
    console.log(a);//报错：Uncaught ReferenceError: a is not defined(…)

    console.log(a);//输出：undefined
var a;

    console.log(a);//输出：undefined  
var a = 10; 
相当于=》
var a;//声明  默认值是undefined
console.log(a);
a = 10;//赋值
  2. 需要注意代码注释中的两个名词——“函数表达式”和“函数声明”。虽然两者都很常用，但是这两者在“准备工作”时，却是两种待遇。
    console.log(f1);     //function f1(){}
function f1(){};     //函数声明

    console.log(f2);         //undefined
var f2=function(){};     //函数表达式
    在“准备工作”中，对待函数表达式就像对待“ var a = 10 ”这样的变量一样，只是声明。而对待函数声明时，却把函数整个赋值了。 总结：“准备工作”中完成的工作有： * 变量、函数表达式--变量声明，默认赋值为undefined； * this--赋值； * 函数声明--赋值； 这三种数据的准备情况我们称之为“执行上下文”或者“执行上下文环境”。 其实，javascript在执行一个代码段之前，都会进行这些“准备工作”来生成执行上下文。这个“代码段”其实分三种情况--全局代码，函数体，eval代码。 
    var a=10;
function fn(){
 console.log(a);      //a是自由变量
}                     //函数创建时，就确定了a要取值的作用域
function bar(f){
 var a=20;
 f();          //打印“10”，而不是“20”
}
bar(fn);
  3. this ：this的取值，分四种情况。（注意：在函数中this到底取何值，是在函数真正被调用执行的时候确定的，函数定义的时候确定不了。因为this的取值是执行上下文环境的一部分，每次调用函数，都会产生一个新的执行上下文环境。）
